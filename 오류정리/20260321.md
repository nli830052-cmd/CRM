## 🏗️ 1. 안드로이드 성능 및 비동기 처리 [해결 완료 ✅]

### ❌ 문제점: 메인 스레드 점유 (UI Freeze)
- 13,000건의 주소록을 로딩하고 전송하는 과정에서 메인(UI) 스레드가 멈춰버리는 현상.

### ✅ 해결 결과: 비동기(Coroutine) & 청크(Chunk) 전송 도입
- **Dispatchers.IO 적용**: 모든 데이터 로드와 전송을 백그라운드 스레드에서 수행 (UI 버벅임 0%).
- **청크(Chunk) 처리**: 13,416명의 데이터를 **100명씩 130번의 패키지로 분할**하여 전송 (속도 및 상호 통신 횟수 비약적 향상).
- **진행률 가시화**: 동기화 중 버튼 안에 **"주소록: 45%"** 처럼 퍼센테이지를 실시간으로 표시.

## 🏗️ 2. 데이터 무결성 및 중복 방지 [해결 완료 ✅]

### ❌ 문제점: UNIQUE 제약 조건 충돌 (Sync Crash)
- 중복된 전화번호 입력 시 `UniqueViolation` 에러가 발생하여 서버가 멈추던 현상.

### ✅ 해결 결과: PostgreSQL UPSERT & 정규화(Normalization)
- **UPSERT (On Conflict) 도입**: `on_conflict_do_nothing` 로직을 서버에 이식하여, 중복된 번호가 들어와도 에러 없이 조용히 넘어가도록 설계.
- **번호 정규화**: 서버와 클라이언트 모두에서 하이픈(`-`), 공백 등을 제거하고 **순수 숫자**로만 저장/매칭하여 검색 및 무결성 확보.

## 🏗️ 3. API 통신 아키텍처 개선 [해결 완료 ✅]

### ❌ 문제점: 비효율적인 개별 전송 방식
- 13,000건의 데이터를 1건씩 따로 쏘던 방식을 그대로 유지하면 아무리 비동기라도 데이터 전송 시간이 너무 길어짐.

### ✅ 해결 결과: 벌크(Bulk) API & 인메모리 매핑(Mapping) 최적화
- **Bulk Insert API**: 한 번의 HTTP 호출로 100건을 한꺼번에 전달하는 `POST /contacts/bulk/` 엔드포인트 구축.
- **고속 매핑(Fast Lookup)**: 통화/문자 기록을 보낼 때마다 서버에 물어보는 대신, 서버 전체 명단을 한 번만 가져와 **폰의 메모리(Map 구조)**에서 즉시 ID를 매칭함 (API 호출 횟수 99% 절감).

## 🏗️ 4. 코틀린 컴파일 오류 및 타입 추론 개선 [해결 완료 ✅]

### ❌ 문제점: `Unresolved reference 'Quad'` & 타입 추론 실패
- `Quad` 데이터 클래스가 올바르게 정의되지 않아 컴파일 에러 발생.
- `forEachIndexed` 람다 식 내에서 `Triple` 과 `Quad` 의 속성(`.first`, `.second` 등)을 컴파일러가 인식하지 못해 **"Cannot infer type"** 오류 발생.

### ✅ 해결 결과: 명시적 데이터 클래스 및 타입 선언
- **`SmsItem` 데이터 클래스 도입**: 모호한 `Quad` 대신 명확한 이름의 `SmsItem`을 최상위 레벨로 분리하여 접근성 및 가독성 확보.
- **명시적 타입 명명**: 람다 인자에 `callTuple: Triple<String, Int, String>` 처럼 정확한 타입을 명시하여 컴파일러의 타입 추론 누락 방지.
- **인덱스 안전성(Column Index Check)**: 컬럼이 존재하지 않을 경우를 대비한 `-1` 체크 로직 추가로 런타임 안정성 보강.
